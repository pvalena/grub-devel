# AI Review: MR !159 - Use FDT bootargs when none specified

1 commit adding fallback to firmware device tree bootargs
when no kernel command line arguments are given to the
`linux` GRUB command.

**Commits:**
1. **605ea449f** - loader:efi:linux: Use device tree bootargs
   when none are specified

## Issues Found

### Issue 1: First FDT bootarg consumed by BOOT_IMAGE=
prefix — missing separator

**Location:** `grub-core/loader/efi/linux.c`,
`grub_cmd_linux()`, lines ~610-622

```c
cmdline_size = sizeof (LINUX_IMAGE) + dt_len;
...
grub_memcpy (linux_args, LINUX_IMAGE, sizeof (LINUX_IMAGE));
grub_memcpy (linux_args + sizeof (LINUX_IMAGE) - 1,
             dt_bootargs, dt_len + 1);
```

`LINUX_IMAGE` is defined as `"BOOT_IMAGE="` (in
`include/grub/lib/cmdline.h`). The code concatenates the
FDT bootargs directly after this prefix, overwriting its
null terminator with no space in between.

If the FDT bootargs are `console=ttyS0 root=/dev/mmcblk0`:
- Produced: `BOOT_IMAGE=console=ttyS0 root=/dev/mmcblk0`
- Expected: `BOOT_IMAGE=/vmlinuz console=ttyS0 root=...`

The kernel parses `BOOT_IMAGE=console=ttyS0` as a single
token, silently consuming the first bootarg as BOOT_IMAGE's
value. On ARM/ARM64 boards using firmware-provided bootargs,
this drops the first parameter (often the console
configuration), causing missing serial output or other
boot failures.

The original code path uses `grub_create_loader_cmdline`
which writes `argv[0]` (the kernel image path) immediately
after `BOOT_IMAGE=`, then spaces between subsequent args.
The DT path should follow the same pattern.

**Draft fix:**

```diff
--- a/grub-core/loader/efi/linux.c
+++ b/grub-core/loader/efi/linux.c
@@ -598,21 +598,25 @@
   {
     char *dt_bootargs = NULL;
-    grub_size_t dt_len = 0;
+    grub_size_t dt_len = 0, img_len;

     err = get_fdt_bootargs (&dt_bootargs);
     if (err)
       goto fail;
     if (!dt_bootargs)
       goto build_default_cmdline;

     dt_len = grub_strlen (dt_bootargs);
+    img_len = grub_strlen (argv[0]);

-    cmdline_size = sizeof (LINUX_IMAGE) + dt_len;
+    cmdline_size = sizeof (LINUX_IMAGE) + img_len
+                   + 1 + dt_len;

     linux_args = grub_malloc (cmdline_size);
     ...

     grub_memcpy (linux_args, LINUX_IMAGE,
                  sizeof (LINUX_IMAGE));
-    grub_memcpy (linux_args + sizeof (LINUX_IMAGE) - 1,
-                 dt_bootargs, dt_len + 1);
+    grub_memcpy (linux_args + sizeof (LINUX_IMAGE) - 1,
+                 argv[0], img_len);
+    linux_args[sizeof (LINUX_IMAGE) - 1 + img_len] = ' ';
+    grub_memcpy (linux_args + sizeof (LINUX_IMAGE)
+                 + img_len, dt_bootargs, dt_len + 1);
```

Produces: `BOOT_IMAGE=/vmlinuz console=ttyS0 root=...`
— matching the format from the original code path.

## Additional findings

In `get_fdt_bootargs`, `len` is `grub_uint32_t` (unsigned)
but checked with `len <= 0`, which is equivalent to
`len == 0` for unsigned types. The behavior is correct
(skip empty bootargs), but a reader might expect a signed
comparison. Using `len == 0` would be clearer.

For more details: https://gitlab.freedesktop.org/pvalena/grub/-/blob/main/reviews/pr159_reasoning.txt
