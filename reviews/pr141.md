# AI Review: MR !141 - Map entire 512G in every replaced PML4 entry

1 commit fixing the x86_64 EFI relocator page table setup to fully
populate PDPT entries when replacing PML4 entries.

**Commits:**
1. **a802d1112** - relocator/x86: map entire 512G in every replaced PML4 entry

## Issues Found

No issues found.

The old code computed `npt2pages` (PD page count) from actual memory
needs, then `npt3pages` (PML4 entries to replace) from that. When a
PML4 entry was replaced, only `npt2pages` PDPT entries were populated
-- the rest were uninitialized, unmapping memory the firmware had
mapped in the 1 GiB-to-512 GiB range under that PML4 slot.

The fix renames the intermediate variable to `npdptentries` (number
of PDPT entries needed for actual memory), computes `npt3pages` from
that (PML4 entries to replace), then sets
`npt2pages = npt3pages << PAGE_IDX_SIZE` so each replaced PML4 entry
gets a full 512 PD pages. The preamble size calculation
(`npt2pages + npt3pages + 1 + 1` pages) and all three page-table
filling loops are consistent with the new counts. The identity-mapped
2 MiB huge page entries beyond actual RAM are harmless -- they cover
physical addresses that nothing accesses.
