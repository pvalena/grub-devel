# AI Review: MR !155 - Native NVMe driver based on SeaBIOS

1 commit adding a PCI NVMe disk driver ported from SeaBIOS
(~786 lines in nvme.c + 208 lines in nvme-int.h).

**Commits:**
1. **97be8c1c3** - Add native NVMe driver based on SeaBIOS

## Issues Found

### Issue 1: No timeout in nvme_wait() — infinite hang on
unresponsive controller

**Location:** `grub-core/disk/nvme.c`, `nvme_wait()`,
lines ~397-412

```c
while (!nvme_poll_cq(sq->cq)) {
    /* FIXME
    yield();
    if (timer_check(to)) {
        warn_timeout();
        return nvme_error_cqe();
    }*/
}
```

The SeaBIOS timer/yield code is commented out but not replaced
with GRUB equivalents. If the controller never posts a
completion (hardware fault, firmware bug, device removed),
GRUB hangs permanently. This function is called on every I/O
operation and every admin command, so any controller hiccup
during boot hangs the bootloader with no recovery.

The same pattern affects `nvme_wait_csts_rdy()` (~lines
537-555): the only exit is a CSTS_FATAL check — a controller
that simply doesn't respond (timeout without fatal flag) also
hangs forever.

**Draft fix:**

```diff
--- a/grub-core/disk/nvme.c
+++ b/grub-core/disk/nvme.c
@@ -397,12 +397,14 @@
 static struct nvme_cqe
 nvme_wait(struct nvme_sq *sq)
 {
-    // static const unsigned nvme_timeout = 5000 /* ms */;
-    // grub_uint32_t to = timer_calc(nvme_timeout);
+    grub_uint64_t endtime = grub_get_time_ms () + 5000;
     while (!nvme_poll_cq(sq->cq)) {
-        /* FIXME
-        yield();
-
-        if (timer_check(to)) {
-            warn_timeout();
+        if (grub_get_time_ms () > endtime) {
+            grub_dprintf("nvme", "nvme_wait timed out\n");
             return nvme_error_cqe();
-        }*/
+        }
     }
```

### Issue 2: Only reads 32-bit BAR — wrong MMIO address on
systems with NVMe above 4 GB

**Location:** `grub-core/disk/nvme.c`,
`grub_nvme_pci_probe()`, lines ~656-658

```c
bar = grub_pci_read (
    grub_pci_make_address (dev, GRUB_PCI_REG_ADDRESS_REG0));
reg = grub_pci_device_map_range (
    dev, bar & GRUB_PCI_ADDR_MEM_MASK, sizeof (*reg));
```

NVMe controllers are required to use a 64-bit memory BAR
(BAR0). Only the lower 32 bits are read here; BAR1 (upper
32 bits) is never fetched. On systems where firmware assigns
the NVMe MMIO range above 4 GB, the upper address bits are
silently lost, and `grub_pci_device_map_range` maps the wrong
physical address — causing reads/writes to hit an unrelated
or unmapped region.

In contrast, the in-tree AHCI driver and PR !156's NVMe driver
both read BAR1 and combine for a full 64-bit address.

**Draft fix:**

```diff
--- a/grub-core/disk/nvme.c
+++ b/grub-core/disk/nvme.c
@@ -653,8 +653,22 @@
-    bar = grub_pci_read (
-        grub_pci_make_address (dev, GRUB_PCI_REG_ADDRESS_REG0));
-    reg = grub_pci_device_map_range (
-        dev, bar & GRUB_PCI_ADDR_MEM_MASK, sizeof (*reg));
+    grub_uint32_t bar_lo, bar_hi;
+    grub_uint64_t bar64;
+
+    bar_lo = grub_pci_read (
+        grub_pci_make_address (dev, GRUB_PCI_REG_ADDRESS_REG0));
+    if ((bar_lo & GRUB_PCI_ADDR_MEM_TYPE_MASK)
+        == GRUB_PCI_ADDR_MEM_TYPE_64)
+      {
+        bar_hi = grub_pci_read (
+            grub_pci_make_address (dev,
+                GRUB_PCI_REG_ADDRESS_REG1));
+        bar64 = ((grub_uint64_t) bar_hi << 32)
+                | (bar_lo & GRUB_PCI_ADDR_MEM_MASK);
+      }
+    else
+      bar64 = bar_lo & GRUB_PCI_ADDR_MEM_MASK;
+
+    reg = grub_pci_device_map_range (dev, bar64, sizeof (*reg));
```

### Issue 3: Missing disk->log_sector_size — wrong LBA
addressing on 4K-sector NVMe drives

**Location:** `grub-core/disk/nvme.c`, `grub_nvme_open()`,
lines ~718-738

```c
disk->total_sectors = ns->lba_count;
disk->max_agglomerate = ns->max_req_size;
disk->id = ns->nsnum;
disk->data = ns;
return 0;
```

`disk->log_sector_size` is never set, so GRUB uses the
default `GRUB_DISK_SECTOR_BITS` (9 = 512 bytes). If the
NVMe namespace uses 4K logical sectors (flbas selects an
LBA format with lbads = 12), GRUB converts byte offsets to
512-byte sector units, but the driver forwards those as LBAs
to the NVMe controller which interprets them as 4K-sector
LBAs. The mismatch causes:

- **Wrong LBA**: byte offset 4096 → GRUB sector 8 → NVMe
  LBA 8 → NVMe byte offset 32768 (should be 4096)
- **Buffer overflow**: `nvme_readwrite` uses
  `buf + i * ns->block_size` (4096-byte stride) with `i`
  counting in GRUB's 512-byte sector units. A request for
  8 GRUB-sectors (4KB) sends count=8 to the NVMe (32KB),
  overflowing the 4KB buffer.

PR !156 correctly sets
`disk->log_sector_size = dev->log_sector_size`.

**Draft fix:**

```diff
--- a/grub-core/disk/nvme.c
+++ b/grub-core/disk/nvme.c
@@ -729,6 +729,17 @@
   disk->total_sectors = ns->lba_count;
   disk->max_agglomerate = ns->max_req_size;
+
+  {
+    grub_uint32_t bs = ns->block_size;
+    int log_size = 0;
+    while (bs > 1) {
+      bs >>= 1;
+      log_size++;
+    }
+    disk->log_sector_size = log_size;
+  }
+
   disk->id = ns->nsnum;
```

## Additional findings

The driver doesn't use `grub_memalign_dma32` /
`grub_dma_get_phys` — it uses plain `grub_memalign` and casts
virtual addresses to physical via `(grub_size_t)`. This works
on identity-mapped platforms (i386-pc, i386-coreboot) but is
not portable to platforms with IOMMUs or non-identity VA/PA
mappings. The SeaBIOS heritage makes this a known design
choice rather than a bug, but limits the driver's portability
compared to PR !156's approach.

`GRUB_MOD_FINI` does not free controller or namespace
structures, nor does it disable the controller — unloading
the module leaks all allocated resources. In-tree drivers like
AHCI register a preboot hook for proper teardown.

For more details: https://gitlab.freedesktop.org/pvalena/grub/-/blob/main/reviews/pr155_reasoning.txt
