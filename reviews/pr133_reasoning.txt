== Issue 1: GRUB_MOD_INIT leaks active DMA device on allocation failure ==

Discovery:
Reading GRUB_MOD_INIT (lines ~970-1010). The initialization flow is:
1. grub_pci_iterate calls grub_virtionet_pci_iter
2. grub_virtionet_pci_iter calls grub_virtio_net_dev_init
3. grub_virtio_net_dev_init maps PCI BARs, sets up virtqueues, writes
   DRIVER_OK to device_status -- device is now actively DMA-capable
4. Back in GRUB_MOD_INIT, txbuf and rcvbuf are allocated

If step 4 fails, the code does:
  grub_free(virtdata);
  grub_print_error();
  return;

Source trace:
At this point, virtdata->pci_common holds the mapped MMIO pointer to
the device's common config. The device is in DRIVER_OK state (set at
line ~498 in grub_virtio_net_dev_init). The RX virtqueue has all 16
descriptors populated (avail.idx = 16 at line 415) pointing to static
rx_buff[] arrays. The device will DMA received packets into those
buffers.

After grub_free(virtdata), the pci_common pointer is lost. The device
continues operating. grub_vionet_card.data is still NULL, so
GRUB_MOD_FINI's check `if (data)` skips cleanup entirely.

Consequence:
The device continues DMA-ing into static BSS memory (rx_buff arrays).
If the module is later unloaded and its BSS reused, the device writes
to memory owned by something else. The PCI mapped ranges (pci_common,
virtio_notify, virtio_net_cfg) are also leaked on i386_ieee1275.

Fix verification:
The fix should reset the device before freeing virtdata:
  virtdata->pci_common->device_status = 0;
  grub_free(virtdata);
This stops the device from DMA-ing. The PCI unmapping issue (Issue 5)
is related but separate.

== Issue 2: TX descriptor reuse without checking used.idx ==

Discovery:
Reading grub_virtionet_write() (lines ~724-762). The TX path:
  grub_uint16_t avail_idx = card->v_tx->avail.idx % GRUB_VIO_Q_SZ;
  // write to desc[avail_idx] unconditionally
  card->v_tx->avail.idx++;

Source trace:
The code never reads v_tx->used.idx. With GRUB_VIO_Q_SZ=16:
- Send 1: desc[0], avail.idx=1
- Send 2: desc[1], avail.idx=2
- ...
- Send 16: desc[15], avail.idx=16
- Send 17: desc[0] again (16 % 16 = 0)

At send 17, desc[0] is overwritten regardless of whether the device
has consumed it (i.e., whether used.idx >= 1). The desc data buffer
at desc[0].addr is a static tx_buff[] array, so the overwrite doesn't
corrupt heap, but the device may be mid-DMA on that buffer.

Consequence:
If 16+ packets are sent rapidly (e.g., TCP retransmit burst, ARP
storm), the driver overwrites a descriptor buffer the device may still
be reading, causing corrupted packet transmission. In GRUB's typical
single-threaded, slow network usage this is unlikely but not impossible
-- HTTP downloads with packet loss can trigger rapid retransmits.

Fix verification:
Before writing to a TX descriptor, poll used.idx to verify the
descriptor has been consumed. A simple spin-wait with timeout:
  while ((avail_idx - v_tx->used.idx) >= GRUB_VIO_Q_SZ) { /* spin */ }
This is the standard approach in minimal virtio drivers.

== Issue 3: RX and TX avail rings never updated ==

Discovery:
Reading grub_vio_q_setup() (lines ~270-288):
  virtq->avail.ring[i] = i;  // initialized to {0,1,...,15}
Then reading grub_virtionet_read() (lines ~821-835): after processing
a received packet with desc_idx from the used ring, the code does:
  card->v_rx->avail.idx++;
but never writes desc_idx into avail.ring[avail.idx % GRUB_VIO_Q_SZ].

Source trace:
The expected virtio driver behavior when making a descriptor available:
  1. Write the descriptor index to avail.ring[avail.idx % queue_size]
  2. Memory barrier
  3. Increment avail.idx
(Based on reading other virtio driver implementations and the virtq
struct layout; not independently confirmed against the spec document.)

The code does step 3 but skips step 1. The avail ring permanently
contains {0, 1, ..., 15}. The device always sees the same sequence
of descriptor indices regardless of which one was just freed.

This works accidentally when:
- The device returns descriptors in order (0, 1, 2, ..., 15, 0, ...)
- avail.idx wraps correctly so avail.ring[idx%16] happens to equal
  the right descriptor index

If the device returns descriptors out of order (e.g., desc 3 before
desc 0), then when the driver "frees" desc 3, avail.ring[next%16]
still says "0" (or whatever the static value is), offering the wrong
descriptor to the device.

Consequence:
With an out-of-order virtio device implementation, the driver would
offer already-in-use descriptors while leaving freed ones stranded.
This causes duplicate buffers and eventual receive buffer exhaustion.
QEMU's virtio-net returns descriptors in order, masking the bug.

Fix verification:
After processing desc_idx, before incrementing avail.idx:
  card->v_rx->avail.ring[card->v_rx->avail.idx % GRUB_VIO_Q_SZ] = desc_idx;
Same fix needed for TX side.

== Issue 4: NULL dereference if preboot hook registration fails ==

Discovery:
Reading GRUB_MOD_INIT (lines ~1005-1008):
  fini_hnd = grub_loader_register_preboot_hook(...);
No NULL check on fini_hnd.

Source trace:
grub_loader_register_preboot_hook (grub-core/commands/boot.c) calls
grub_malloc internally. On failure it returns NULL. In GRUB_MOD_FINI:
  grub_loader_unregister_preboot_hook(fini_hnd);
This function accesses fini_hnd->next and fini_hnd->prev without
NULL guards.

Consequence:
If the preboot hook registration fails (OOM) and the module is later
unloaded, grub_loader_unregister_preboot_hook dereferences NULL,
crashing GRUB.

Fix verification:
Add a NULL check: `if (fini_hnd == NULL) { grub_print_error(); }`.
In GRUB_MOD_FINI: `if (fini_hnd) grub_loader_unregister_preboot_hook(...)`.

Note: An earlier draft of this review included an Issue 5 claiming PCI
mapped ranges were never unmapped during teardown, stating that
grub_pci_device_unmap_range() on i386_ieee1275 "performs actual resource
cleanup." This was removed after reading the actual implementation at
grub-core/bus/i386/ieee1275/pci.c:38 -- the function is an empty no-op.
The inline version in include/grub/i386/pci.h:84 is also a no-op. Not
calling an empty function is not a bug.
