## Issues Found

### Issue 1: Admin queue state not reset in restore_hw

**Location:** `grub-core/disk/nvme.c`, `grub_nvme_restore_hw()`,
lines ~490-524

After `fini_hw` disables the controller (`controller_config = 0`),
the admin queue indices (`queue[ads].idx`, `queue[adc].idx`) and
phase tracking (`queue[adc].round`) retain their values from the
2 delete commands issued during shutdown. When `restore_hw`
re-enables the controller, the controller resets its internal SQ
tail and CQ head to 0, but the host's indices are stale.

Concrete scenario (pciinit issues 3 admin commands, fini_hw issues
2 more = 5 total): after fini_hw, admin SQ idx=1, CQ idx=1. In
restore_hw:

- **SQ hang**: Host writes command to slot 1, advances idx to 0,
  rings doorbell with value 0. Controller's internal tail is also
  0 (from re-enable), sees no advancement (0→0), does not fetch.
- **CQ hang**: Host polls entry 1, but controller writes
  completion to entry 0 (its internal position after re-enable).
  Host never sees it.

Not a straightforward fix. `restore_hw` must reset admin queue
state before sending commands. At minimum:
```c
(*pdev)->queue[ads].idx = 0;
(*pdev)->queue[adc].idx = 0;
(*pdev)->queue[adc].round = 0;
grub_memset((void *)grub_dma_get_virt((*pdev)->cq_buffer),
            0, NVME_CQ_ENTRY_SIZE * NVME_QUEUE_SIZE);
```
The CQ buffer must be zeroed so all phase bits are 0, matching
round=0. The SQ buffer should also be re-zeroed for cleanliness.
AQA/ASQ/ACQ registers retain their values across disable/enable
(no controller reset), so they don't need reprogramming.

For more details:
[pr156_reasoning.txt](https://gitlab.freedesktop.org/pvalena/grub/-/blob/main/reviews/pr156_reasoning.txt)

## Additional findings

Previous issues 1-3 are all fixed. Byte-order conversions
(`grub_cpu_to_le32`, `grub_le_to_cpu32/64`) added to doorbell
writes, command fields, register reads, and completion parsing.
Error paths now properly free all DMA buffers via a unified `fail`
label. Delete queue functions now also free their DMA buffers and
NULL the pointers.

Three `cap_dstrd` reads in `create_admin_queues`,
`create_io_submission_queue`, and `create_io_completion_queue`
use `(nvme->regs->cap >> 32) & 0xf` without `grub_le_to_cpu64`.
This would give wrong doorbell offsets on big-endian platforms.
Not reported as a separate issue because the driver has other
BE limitations (virt==phys assumption, PRP list entries written
without LE conversion).
