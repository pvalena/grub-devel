# AI Review: MR !156 - NVMe native disk support

1 commit adding a PCI NVMe disk driver (~649 lines) with
DMA-safe allocations and preboot hooks.

**Commits:**
1. **af11821d1** - Implement NVMe native disk support

## Issues Found

### Issue 1: Wrong opcode and QID in delete_io_completion_queue

**Location:** `grub-core/disk/nvme.c`,
`delete_io_completion_queue()`, lines ~298-304

```c
const struct nvme_s_queue_entry e = {
    .dw[0]  = 1,      /* opcode: should be 0x04 */
    .dw[10] = ioc,    /* QID: should be ioc >> 1 */
};
```

Two bugs in one command:

**Opcode**: `dw[0] = 1` is opcode 0x01 = Create I/O
Submission Queue. Delete I/O Completion Queue is opcode
0x04. The controller receives a create command when a
delete was intended.

**QID**: `dw[10] = ioc` passes the raw enum value 3 as the
queue identifier. But `create_io_completion_queue` used
`ioc >> 1` = 1 as the QID when creating the queue. The
delete targets a queue that was never created.

The same QID bug affects `delete_io_submission_queue`:
`dw[10] = ios` passes 2, but the queue was created with
QID `ios >> 1` = 1. (Its opcode 0x00 is correct though.)

Both functions are called from `grub_nvme_fini_hw`, which
then disables the controller via `controller_config = 0`.
The controller reset implicitly invalidates all queues, so
the malformed delete commands are masked in practice.

**Draft fix:**

```diff
--- a/grub-core/disk/nvme.c
+++ b/grub-core/disk/nvme.c
@@ -280,7 +280,7 @@
 static int delete_io_submission_queue(...)
 {
 	const struct nvme_s_queue_entry e = {
 		.dw[0]  = 0,
-		.dw[10] = ios,
+		.dw[10] = ios >> 1,
 	};
 ...
 static int delete_io_completion_queue(...)
 {
 	const struct nvme_s_queue_entry e = {
-		.dw[0]  = 1,
-		.dw[10] = ioc,
+		.dw[0]  = 4,
+		.dw[10] = ioc >> 1,
 	};
```

### Issue 2: Unchecked return values from IO queue creation
in grub_nvme_pciinit

**Location:** `grub-core/disk/nvme.c`,
`grub_nvme_pciinit()`, lines ~432-433

```c
create_io_completion_queue(nvmedev);
create_io_submission_queue(nvmedev);

if(identify(nvmedev)) {
    grub_free (nvmedev);
```

Both `create_io_completion_queue` and
`create_io_submission_queue` return non-zero on failure,
but the return values are ignored. If allocation or the
admin command fails, the queue state remains uninitialised.
The subsequent `identify()` call invokes `nvme_cmd()` on the
IO admin queue, which succeeds or fails independently, but
if it succeeds the device is added to the list with broken
IO queues. Any later disk read would use the uninitialised
IO queue state — `nvme_cmd` would write a command to a NULL
or garbage base pointer.

**Draft fix:**

```diff
--- a/grub-core/disk/nvme.c
+++ b/grub-core/disk/nvme.c
@@ -432,8 +432,14 @@
-  create_io_completion_queue(nvmedev);
-  create_io_submission_queue(nvmedev);
+  if (create_io_completion_queue(nvmedev))
+    {
+      grub_free (nvmedev);
+      return 0;
+    }
+  if (create_io_submission_queue(nvmedev))
+    {
+      grub_free (nvmedev);
+      return 0;
+    }
```

### Issue 3: grub_nvme_restore_hw doesn't wait for
controller ready

**Location:** `grub-core/disk/nvme.c`,
`grub_nvme_restore_hw()`, lines ~466-476

```c
(*pdev)->regs->controller_config = NVME_CC_EN | ...;
create_io_completion_queue(*pdev);
create_io_submission_queue(*pdev);
```

After setting `controller_config` to enable the controller,
the function immediately creates IO queues without waiting
for `CSTS.RDY` to become 1. The NVMe spec requires the host
to wait for the controller to signal readiness before
submitting admin commands. The queue creation sends admin
commands (Create IO CQ / Create IO SQ) to a controller that
hasn't finished its enable sequence — these commands will
be silently dropped or cause undefined behavior.

`grub_nvme_pciinit` correctly polls `controller_status`
in a loop before proceeding. `restore_hw` skips this.

Not a straightforward fix. The restore path needs the same
CSTS.RDY polling loop that `grub_nvme_pciinit` has (lines
~399-415), plus the admin queue must be re-initialised
before sending create commands (the AQA/ASQ/ACQ registers
need to be reprogrammed after a controller reset).

## Additional findings

Resource leaks on the `identify()` failure path: when
`identify(nvmedev)` fails, only `grub_free(nvmedev)` is
called. The DMA buffers (`prp_list`, `ios_buffer`,
`ioc_buffer`, `sq_buffer`, `cq_buffer`) are not freed, and
the controller is not disabled. The same leak pattern occurs
on the unchecked queue creation failures from Issue 2.

For more details: https://gitlab.freedesktop.org/pvalena/grub/-/blob/main/reviews/pr156_reasoning.txt
