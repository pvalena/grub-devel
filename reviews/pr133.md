## Issues Found

### Issue 1: `GRUB_MOD_INIT` leaks active DMA device on allocation failure

**Location:** `grub-core/net/drivers/virtio/virtionet.c`, `GRUB_MOD_INIT`,
lines ~983-998

When `grub_malloc` for `txbuf` or `rcvbuf` fails, the code frees
`virtdata` and returns. At this point, the virtio device has already
been fully initialized via `grub_virtio_net_dev_init` and is in
`DRIVER_OK` state -- actively DMA-capable. Freeing `virtdata` loses
the `pci_common` pointer needed to reset the device. The PCI mapped
ranges (`pci_common`, `virtio_notify`, `virtio_net_cfg`) are also
leaked. Since `grub_vionet_card.data` is never set, `GRUB_MOD_FINI`
will not attempt cleanup.

```c
grub_vionet_card.txbuf = grub_malloc (grub_vionet_card.txbufsize);
if (!grub_vionet_card.txbuf)
  {
    grub_free (virtdata);     // device still in DRIVER_OK, DMA active
    grub_print_error ();
    return;
  }
```

The fix should reset the device (write 0 to device_status via
`pci_common`) before freeing `virtdata`.

### Issue 2: TX path never waits for device completion before reusing descriptors

**Location:** `grub-core/net/drivers/virtio/virtionet.c`,
`grub_virtionet_write()`, lines ~738-762

The TX path selects the next descriptor via
`avail.idx % GRUB_VIO_Q_SZ` and writes to it unconditionally, never
checking `used.idx` to verify the device has finished with the
descriptor. With `GRUB_VIO_Q_SZ = 16`, the 17th transmit overwrites
descriptor 0 regardless of whether the device has consumed it.

```c
grub_uint16_t avail_idx = card->v_tx->avail.idx % GRUB_VIO_Q_SZ;
grub_addr_t tx_q_addr = (grub_addr_t)(card->v_tx->desc[avail_idx].addr);
// ... overwrites descriptor without checking used.idx
```

The fix should poll `used.idx` before reusing a descriptor, or at
minimum track which descriptors are in-flight.

### Issue 3: RX and TX avail rings are never updated after initialization

**Location:** `grub-core/net/drivers/virtio/virtionet.c`,
`grub_virtionet_read()` (line ~832), `grub_virtionet_write()` (line ~756)

The `avail.ring[]` array is initialized to `{0, 1, 2, ..., 15}` in
`grub_vio_q_setup` and never updated. The expected virtio driver
behavior is to write the descriptor index to
`avail.ring[avail.idx % queue_size]` before incrementing `avail.idx`.
The RX path increments `avail.idx` after processing a
received packet but does not write the freed `desc_idx` back into
the avail ring.

This works only because QEMU returns used descriptors in order and
the static ring values match. If the device returns descriptors out
of order (permitted by the spec), the wrong descriptor would be
offered, leading to buffer duplication or data loss.

### Issue 4: NULL dereference if `grub_loader_register_preboot_hook` fails

**Location:** `grub-core/net/drivers/virtio/virtionet.c`,
`GRUB_MOD_INIT` (line ~1007), `GRUB_MOD_FINI` (line ~1019)

`grub_loader_register_preboot_hook` returns NULL on allocation failure.
The return value is stored in `fini_hnd` without a NULL check. In
`GRUB_MOD_FINI`, `grub_loader_unregister_preboot_hook(fini_hnd)` is
called unconditionally, which dereferences `fini_hnd->next` and
crashes.

For more details: https://github.com/pvalena/grub-devel/blob/main/reviews/pr133_reasoning.txt
