PR176 Investigation — libgcrypt 1.11 → 1.12.2 with MLDSA
==========================================================

3 commits, 178 files (~24k lines). Adds Dilithium/ML-DSA post-quantum
signature verification to GRUB.

Focus: every line of GRUB-specific code (patches 17-20, import script,
crypto.h, Makefile, test module). Dilithium verify path read line by
line. Upstream cipher/hash bulk changes scanned for anomalies.

1. Dilithium verify path (security-critical)
---------------------------------------------
Read dilithium.c:dilithium_verify() (lines 218-270) and
dilithium-dep.c:crypto_sign_verify_internal() (lines 1135-1210).

Verify flow:
  dilithium_verify() builds the prefix 'pre' from ctx/ctxlen →
  calls crypto_sign_verify_internal_N() (N = 2/3/5 for the three
  security levels, generated by three #include "dilithium-dep.c"
  passes with different DILITHIUM_MODE defines).

crypto_sign_verify_internal() does:
  1. Check siglen == CRYPTO_BYTES
  2. Unpack public key (rho, t1) and signature (c, z, h)
  3. Check z norm < GAMMA1 - BETA
  4. Compute mu = SHAKE256(H(pk) || pre || msg)
     ^^^^ 'pre' is absorbed here — its presence/absence changes mu
  5. Matrix-vector multiply: w1 = Az - c*2^d*t1
  6. Reconstruct w1 using hint h
  7. Compute c2 = SHAKE256(mu || pack_w1(w1))
  8. Compare c == c2 (constant-time via loop)

The pre buffer matters: if prelen==0, the SHAKE256 absorb of pre
is a no-op. If prelen==2, two bytes {0,0} are absorbed, changing mu,
which cascades to a different c2 value. So signatures generated with
the prefix cannot be verified without it, and vice versa.

2. Patch 18_dilithium — FIPS 204 compliance fix
-------------------------------------------------
Upstream dilithium_verify: when ctx==NULL && ctxlen==-1, sets
prelen=0 (no domain separator at all).

GRUB patch changes this to: pre={0,0}, prelen=2.

FIPS 204 (ML-DSA) specification requires the 2-byte domain separator
(mode=0, ctxlen=0) even for empty context. The upstream code's
prelen=0 path would reject valid FIPS 204 signatures.

Verified against the test vectors: the test calls verify with both
with_ctx=1 (label="tests", prelen=7: {0,5,'t','e','s','t','s'}) and
with_ctx=0 (no-prefix flag, which maps to ctxlen=-1 in the internal
API). Both test paths must pass with the patch applied, meaning the
sig*_without_ctx test vectors were signed with the {0,0} prefix per
FIPS 204, not without any prefix.

Also verified: 'log_fatal' → '_gcry_fatal_error' in shake128_init
and shake256_init. log_fatal is a libgpg-error function. Both calls
are in fatal error paths (md_open failure) where crashing is the
correct behavior.

3. Patch 17_md — hash_buffers fast-path disable
-------------------------------------------------
Two changes to md.c:

(a) (void)key; (void)keylen; in md_setkey when !USE_BLAKE2:
The switch statement in md_setkey only has cases for BLAKE2 algorithms
that use key/keylen. When BLAKE2 is disabled, the parameters are
unused. The casts suppress -Wunused-parameter.

(b) #if 0 around hash_buffers fast-path in
_gcry_md_hash_buffers_extract:
The import script strips per-algorithm _gcry_*_hash_buffers functions
(they're in the skip regex). Without this patch, spec->hash_buffers
would be NULL for all algorithms, and the conditional
'if (!hmac && spec->hash_buffers)' would never be true. So the
#if 0 is technically redundant — but it's defensive. If a future
import accidentally includes a hash_buffers implementation, the NULL
check would pass and the function would be called. The #if 0 prevents
this.

Actually, re-checking: the skip regex skips functions matching
'_gcry_[a-z0-9_]*_hash_buffers'. But if spec->hash_buffers is set
to a non-NULL function pointer in the spec struct initialization
(which the import script might not strip), calling it would work —
but only if the function body is also present. Since the import
strips the body, spec->hash_buffers should remain NULL (set to 0
in the spec initializer). The #if 0 is a safety belt.

4. Patch 19_mpi — mpih-pow.c header and gpgrt
-----------------------------------------------
(a) _gcry_mpih_add_1_lli declaration: this function is defined in the
new mpih-pow.c (added to the Makefile). Without the declaration,
callers in the same compilation unit would get an implicit declaration
warning or error. Verified the function exists in the imported
mpih-pow.c.

(b) gpgrt_annotate_leaked_object(a) disabled: this is a leak-tracking
helper from libgpg-runtime, not available in GRUB. The 'return'
statement after it is preserved — constant MPIs (flags & 32) are
intentionally never freed.

5. Patch 20_sex — errno in hex2buffer
---------------------------------------
gpg_err_set_errno(EINVAL) disabled in _gcry_hex2buffer. GRUB's
environment doesn't provide this function. Error is still signaled
by the NULL return value. Callers check the return, not errno.

6. Import script changes (import_gcry.py)
------------------------------------------
Read every changed line:

(a) Extra files list: dilithium-common.c, dilithium-dep.c, dilithium.c
added. These are the three Dilithium source files needed for the
pubkey module.

(b) Skip regex for unused files: mceliece6688128f.sh added. This is a
shell script for McEliece key generation — not needed in GRUB.

(c) Skip regex for platform-optimized: 7 new entries for RISC-V and
AArch64 VP cipher variants. GRUB uses generic C implementations.

(d) SIZE_MAX for kdf.c: New libgcrypt 1.12.2 kdf.c uses SIZE_MAX.
GRUB's stddef.h may not define it. The #ifndef guard is correct.

(e) Function skip list changes:
- 'run_selftests' removed from the comma-stripping regex (no longer
  just stripped from struct init — now defined as 0 macro)
- 'md_extract' and '_gcry_md_extract' REMOVED from skip list —
  intentional, needed for SHAKE XOF extraction
- '_gcry_md_get ' (trailing space) → '_gcry_md_get' (no space) —
  broadens the match. Compensated by negative checks:
  'not re.match("_gcry_md_hash_buffers", line)' and
  'not re.match("_gcry_md_get_algo_dlen", line)' ensure these two
  functions are NOT skipped.
  Checked: _gcry_md_get_algo (returns algo ID from handle) IS still
  skipped — matches '_gcry_md_get' prefix AND is in the skip list
  separately. This is intentional (same as before the change).
- New skip entries: mldsa_generate, mldsa_sign, randombytes — GRUB
  only verifies, never generates or signs.
- New #define stubs: 'mldsa_generate 0', 'mldsa_sign 0',
  'randombytes 0' — same pattern as existing dsa_generate, rsa_sign
  stubs.
- 'run_selftests' gets '#define run_selftests 0' instead of being
  stripped. This is needed because the new gcry_pk_spec struct has a
  'selftest' field that may reference run_selftests — defining it as
  0 (NULL) provides a valid initializer.

(f) re.match note: re.match anchors at the start of the line, so
'_gcry_md_get' matches any line starting with that string (including
_gcry_md_get_algo, _gcry_md_get_algo_dlen). The negative checks
prevent false-positive skipping of the needed functions.

7. crypto.h changes
--------------------
Read every addition:

- GPG_ERR_BAD_PUBKEY: new error code for invalid public key format
- selftest_func_t: function pointer type for self-test callbacks.
  Added to both gcry_md_spec and gcry_pk_spec structs.
- gcry_md_extract_t: properly typed function pointer (was void *).
  The extract field in gcry_md_spec changes from 'void *extract' to
  'gcry_md_extract_t extract' — type-safe.
- PUBKEY_FLAG_BYTE_STRING (1<<18): ML-DSA uses raw byte strings for
  keys/signatures, not MPI encoding like RSA/DSA/ECDSA.
- PUBKEY_FLAG_NO_PREFIX (1<<19): skip the ML-DSA domain separator
  prefix (used by the test's without-context path).
- rnd/rndlen in pk_encoding_ctx: for deterministic signatures
  (random-override). GRUB only verifies, so these fields are unused
  but needed for struct layout compatibility.
- grub_crypto_pk_mldsa extern: the pubkey spec for ML-DSA, analogous
  to grub_crypto_pk_rsa, grub_crypto_pk_ecdsa.

9. Test module (mldsa_sexp_test.c)
-----------------------------------
Read every line:

- Uses extern gcry_pk_spec_t _gcry_pubkey_spec_mldsa for direct
  internal API access
- mldsa_verify_signature() builds three S-expressions:
  s_data: (data(flags raw [no-prefix])(value MSG)[(label CTX)])
  s_pubkey: (ALGO(p PUBKEY))
  s_sig: (sig-val(ALGO(s SIG)))
- Calls _gcry_pubkey_spec_mldsa.verify(s_sig, s_data, s_pubkey)
- Each grub_test_assert checks rc == 0
- No memory leaks: S-expressions are not freed — this is a GRUB
  functional test that runs in the shell and doesn't need cleanup.
  (GRUB tests typically don't free resources.)

Test coverage:
- MLDSA-44 (dilithium2), MLDSA-65 (dilithium3), MLDSA-87 (dilithium5)
- Each tested with context ("tests") and without
- Total: 6 verify calls, all must succeed

The test vectors in mldsa_pubkeys.h (~500 lines) and
mldsa_signatures.h (~1750 lines) contain pre-generated keys and
signatures. These are constants — no runtime generation.
