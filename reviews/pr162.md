## Issues Found

### Issue 1: pow with exponent 0 returns 0 instead of 1

**Location:** `grub-core/commands/calc.c`,
`grub_cmd_calc()`, lines ~243-248

```c
case CALC_OP_POW:
    if (num2 == 0)
    {  if (num1 == 0)
          return( grub_error(...) );
       num1 = 0;
       break;
    };
```

For any nonzero x, x^0 = 1. The code sets `num1 = 0`,
so `calc 5 pow 0` returns 0 instead of 1. The loop
below this block already handles the case correctly
(`num3 = 1; for ... num3 *= num1` — with 0 iterations,
`num3` stays 1), but the early return prevents it from
being reached.

**Draft fix:**

```diff
--- a/grub-core/commands/calc.c
+++ b/grub-core/commands/calc.c
@@ -245,7 +245,7 @@
              if (num2 == 0)
              {  if (num1 == 0)
                    return( grub_error(...) );
-                num1 = 0;
+                num1 = 1;
                 break;
              };
```

## Additional findings

Direct output (`grub_printf("%s", res)` at line ~269)
does not print a trailing newline. In an interactive GRUB
session, `calc 1 add 3` would display `4grub>` with the
result and prompt on the same line. The documentation
examples show the result on its own line. Other GRUB
commands that print to the console (e.g., `echo`) add a
trailing newline explicitly.

The `neg` alias for `not` (bitwise NOT) in the docs and
operation table is potentially confusing — `neg` commonly
means arithmetic negation (multiply by -1), not bitwise
complement. The user can use `0 sub N` for arithmetic
negation.
