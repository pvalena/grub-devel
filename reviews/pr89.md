# AI Review: MR !89 - grub-mkconfig: Add access levels for authentication support

3 commit(s) adding access level-based authentication support to grub-mkconfig, allowing
password-protected boot entries with configurable privilege levels per user and entry type.

**Commits:**
1. **02e217411** - grub-mkconfig: Add access levels for authentication support
2. **ee97ae060** - grub-mkconfig: Use access levels in all default grub.d scripts
3. **4848c55b9** - docs: Add access level documentation

## Issues Found

### 1. Dead code: empty password type case unreachable in 00_header.in

**File:** `util/grub.d/00_header.in`, case statement around line 68-77

The case statement checking `"x$pw_user_passtype"` has a `x)` branch intended to catch
empty password types, but it is placed after the `x*)` wildcard branch. In shell `case`
statements, `x*` matches "x" followed by zero or more characters, which includes the bare
"x" (when `$pw_user_passtype` is empty). The `x)` branch is therefore dead code and can
never execute. An empty password type will be reported as "Unrecognized password type"
instead of the intended "No password type specified" message.

### 2. Documentation/code mismatch: field separator in GRUB_USERS

**File:** `docs/grub.texi` vs `util/grub.d/00_header.in` and `util/grub-mkconfig_lib.in`

The documentation states that user definition fields are "separated by semicolons" and
shows the example `root;0;pbkdf2;grub.pbkdf2.sha512.10000.BIGLONGSTRING`. However, the
code uses `cut -d :` (colon delimiter) in both `00_header.in` and `grub-mkconfig_lib.in`.
A user following the documentation will produce a completely non-functional configuration:
the username would be parsed as the entire semicolon-delimited string, the access level
would be empty, and password type detection would fail.

### 3. Multiple non-superuser users silently dropped from --users list

**File:** `util/grub-mkconfig_lib.in`, function `grub_get_auth_user_list`

When multiple non-superuser users match a given access level, the function outputs
`--users user1 user2` (space-separated, unquoted). GRUB's menuentry parser treats
`--users` as taking a single string argument. Without quoting or comma separation, only
`user1` is captured as the `--users` argument; `user2` becomes a stray positional argument
to menuentry. The documented format is `--users=user1,user2` (comma-separated). Any
configuration with two or more non-superuser users at the same access level will silently
grant only the first user access, with no error or warning.

## Review Result

Three issues found. Issue 2 (documentation/code delimiter mismatch) is critical: any user
following the official documentation will produce a broken GRUB configuration. Issue 3
(multi-user list formatting) is a correctness bug affecting non-trivial multi-user setups.
Issue 1 (dead code branch ordering) is a minor defect producing a misleading error message.
