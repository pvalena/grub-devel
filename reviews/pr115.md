# AI Review: MR !115 - util/grub-install: EFI, skip registration when already registered

1 commit. Replaces `grub_install_remove_efi_entries_by_distributor()` with
`grub_install_efi_is_registered()`, which checks whether an EFI boot entry
matching the file path already exists. If found, registration is skipped
instead of removing and re-creating entries. This prevents accidental removal
of similarly named entries (e.g., UKI entries).

**Commits:**
1. **fe3472486** - EFI: skip registration when already registered

## Issues Found

### 1. Error from `grub_install_efi_is_registered()` silently skips registration

**File:** `grub-core/osdep/unix/platform.c`
**Location:** `grub_install_register_efi()`, lines ~154-156

When `grub_install_efi_is_registered()` fails (e.g., `grub_util_exec_pipe()`
returns 0, or `fdopen()` fails), it returns `errno` (a non-zero value). The
caller treats any non-zero return as "already registered":

```c
  ret = grub_install_efi_is_registered (efifile_path);
  if (ret)
    return 0;
```

This means that if `efibootmgr` cannot be executed or the pipe setup fails,
the EFI boot entry will silently not be created, and `grub_install_register_efi()`
returns 0 (success) to its own caller, completely hiding the failure.

In the old code, the same error paths returned `errno`, but the caller propagated
it (`if (ret) return ret;`), so errors were not silently swallowed.

**Fix:** `grub_install_efi_is_registered()` should use distinct return values
for "registered" vs "error", or the error paths should return 0 (not registered)
so that registration proceeds on a best-effort basis, since the version check
for `efibootmgr` earlier in `grub_install_register_efi()` already guards against
`efibootmgr` being missing entirely.

## Review Result

One correctness issue found. The error handling mismatch between the new
`grub_install_efi_is_registered()` return values and the caller's interpretation
means pipe/fdopen failures silently prevent EFI boot entry creation while
reporting success.
