# AI Review: MR !115 - util/grub-install: EFI, skip registration when already registered

1 commit. Replaces `grub_install_remove_efi_entries_by_distributor()` with
`grub_install_efi_is_registered()`, which checks whether an EFI boot entry
matching the file path already exists. If found, registration is skipped
instead of removing and re-creating entries. This prevents accidental removal
of similarly named entries (e.g., UKI entries).

**Commits:**
1. **fe3472486** - EFI: skip registration when already registered

## Issues Found

### 1. Error from `grub_install_efi_is_registered()` silently skips registration

**File:** `grub-core/osdep/unix/platform.c`
**Location:** `grub_install_register_efi()`, lines ~154-156

When `grub_install_efi_is_registered()` fails (e.g., `grub_util_exec_pipe()`
returns 0, or `fdopen()` fails), it returns `errno` (a non-zero value). The
caller treats any non-zero return as "already registered":

```c
  ret = grub_install_efi_is_registered (efifile_path);
  if (ret)
    return 0;
```

This means that if `efibootmgr` cannot be executed or the pipe setup fails,
the EFI boot entry will silently not be created, and `grub_install_register_efi()`
returns 0 (success) to its own caller, completely hiding the failure.

In the old code, the same error paths returned `errno`, but the caller propagated
it (`if (ret) return ret;`), so errors were not silently swallowed.

Not a straightforward fix. The function uses a single `int` return to signal three
distinct states: not registered (0), registered (1), and error (errno). The caller
conflates the latter two. Fixing this properly requires choosing a convention --
e.g., returning -1 for error and adjusting the caller to handle three cases, or
using an out-parameter for the "registered" flag, or restructuring so errors fall
through to best-effort registration. Each option changes the API contract and
needs consideration of how the caller should behave on transient failures (retry?
warn and proceed? abort?). The author should decide the intended failure mode.

### 2. Missing `fclose(fp)` in `grub_install_efi_is_registered()`

**File:** `grub-core/osdep/unix/platform.c`
**Location:** `grub_install_efi_is_registered()`, before `return rc`

The function opens a FILE stream with `fdopen(fd, "r")` but never closes it.
After the `while` loop and `free(line)`, the function returns `rc` without
calling `fclose(fp)`. This leaks the FILE stream and the underlying file
descriptor on every call. Compare with `get_ofpathname()` in the same file,
which properly calls `fclose(fp)` before returning.

**Draft fix** -- add `fclose(fp)` before the return:

```diff
--- a/grub-core/osdep/unix/platform.c
+++ b/grub-core/osdep/unix/platform.c
@@ -128,5 +128,6 @@
     }
 
   free (line);
+  fclose (fp);
   return rc;
 }
```

## Review Result

Two issues found. Issue 1 (error/registered conflation) is a correctness
bug where pipe or fdopen failures silently skip EFI boot entry creation
while reporting success. Issue 2 (missing fclose) is a resource leak on
every call to the function.
