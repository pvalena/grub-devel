# AI Review: MR !145 - Disk cache anti-thrashing and statistics fixes

3 commits fixing disk cache correctness and adding an
anti-thrashing mechanism for hash-collision-heavy workloads.

**Commits:**
1. **65ac2b0fe** - cacheinfo: fix hit ratio calculation and statistics output
2. **8f897b7e0** - disk: fix cache lock and hit counter for invalidated cache entries
3. **b27c5e1a8** - disk: reduce cache slot thrashing

## Issues Found

No issues found.

**Commit 1** fixes two bugs in `cacheinfo.c`: the `hits * 10000`
product overflows on 32-bit builds (now uses `grub_divmod64()`
with 64-bit operands), and the `grub_printf_` argument order was
wrong (was `ratio/100, ratio%100, hits, misses`; format string
expects `hits, ratio/100, ratio%100, misses`). Both fixes are
straightforward and correct.

**Commit 2** fixes `grub_disk_cache_fetch()` to not lock or
count as a hit when a matching cache entry has `data == NULL`
(invalidated). Previously, such entries were locked permanently
(lock set to 1, never cleared because unlock only fires after
the caller uses the returned non-NULL pointer) and incorrectly
counted as hits. The fix guards both `cache->lock = 1` and
the hit counter with `if (cache->data)`, and adds an else-branch
for `grub_disk_cache_misses++`. Also adapts fetch and unlock to
traverse the overflow chain introduced in commit 3.

**Commit 3** adds per-bucket overflow chains to
`grub_disk_cache_table`. When a primary slot is evicted, the
displaced entry's metadata (sector, dev_id, disk_id — but not
data) is saved in a dynamically allocated node prepended to
`cache->next`. On subsequent `grub_disk_cache_store()`, if the
incoming sector is found in the chain (indicating thrashing),
the entry is refilled in-place rather than evicting the primary
slot again. Chain length is bounded to `GRUB_CACHE_EVICTED_MAX`
(3) historical entries plus the current eviction.
`grub_disk_cache_invalidate()` and `grub_disk_cache_invalidate_all()`
correctly walk and free overflow chain nodes. Traced the
store→fetch→unlock→invalidate lifecycle through concrete
scenarios including: ghost entry fetch returning NULL (correct
miss), anti-thrashing refill of a chain entry, and invalidation
of both primary and overflow entries. The `grub_errno` clearing
after `grub_calloc` is intentional — prevents a non-critical
allocation failure from propagating as an error.
