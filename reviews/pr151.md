# AI Review: MR !151 - EFI TPCM module support

2 commits adding a TPCM (Trusted Platform Control Module)
verifier module for x86_64 EFI, interfacing with a C2P
firmware protocol for measurement and control.

**Commits:**
1. **e4527482e** - efi/tpcm: Add UEFI interface for TPCM module
2. **a4df14a27** - efi/tpcm: Add complete support of TPCM module

## Issues Found

### Issue 1: NULL dereference in grub_tpcm_verify_string

**Location:** `grub-core/commands/tpcm.c`,
`grub_tpcm_verify_string()`, lines ~52-70

The `switch (type)` has no `default` case. If `type` doesn't
match one of the three `grub_verify_string_type` values,
`prefix` remains NULL. The subsequent
`grub_strlen(prefix)` dereferences NULL — crash.

The enum currently has exactly three values (all handled),
but the function provides no guard against future extension.
Since this is a security verifier called from the framework,
robustness matters.

**Draft fix:**

```diff
--- a/grub-core/commands/tpcm.c
+++ b/grub-core/commands/tpcm.c
@@ -64,6 +64,8 @@
     case GRUB_VERIFY_COMMAND:
       prefix = "grub_cmd: ";
       break;
+    default:
+      return GRUB_ERR_NONE;
     }
```

### Issue 2: Missing NULL check on strchr in
grub_tpcm_measure_memory

**Location:** `grub-core/commands/efi/tpcm.c`,
`grub_tpcm_measure_memory()`, lines ~146-155

```c
p = grub_strchr (p_context, '|');
p_desc = p + 1;
grub_memcpy (tmp, p_context, (p - p_context));
```

If `p_context` contains no `'|'`, `grub_strchr` returns
NULL. Then `p - p_context` underflows to a huge value and
`grub_memcpy` reads far past the source buffer. `p_desc`
becomes `(void*)1`, crashing in the subsequent
`grub_strlen(desc)` inside `measure_memory()`.

Current callers always include `'|'` (built via
`grub_snprintf("%d|%s", ...)`), but the function is
exported in the header and has no contract enforcement.

**Draft fix:**

```diff
--- a/grub-core/commands/efi/tpcm.c
+++ b/grub-core/commands/efi/tpcm.c
@@ -146,6 +146,9 @@
     p = grub_strchr (p_context, '|');
+    if (p == NULL)
+      return GRUB_ERR_BUG;
+
     p_desc = p + 1;
```

## Additional findings

`update_measured_id()` is called even when
`get_tpcm_stage()` returns `STAGE_INVALID` — the counter
increments but no measurement runs, and `measure_memory`
returns `GRUB_ERR_BAD_OS` (the default, never overwritten).
After 1000 calls (stages 2000-2999 exhausted), all
subsequent file loads through this verifier fail. Not
practically reachable in normal GRUB usage (~10-20
measurements per boot), but moving the `update_measured_id`
call inside the `if (stage != STAGE_INVALID)` block would
be more correct.

The `while (1) { grub_error(...); asm volatile ("hlt"); }`
on verification failure is an intentional hard-stop — halts
the CPU to prevent booting unverified code.

For more details: https://gitlab.freedesktop.org/pvalena/grub/-/blob/main/reviews/pr151_reasoning.txt
