## Issues Found

### Issue 1: Missing fopen NULL check causes crash

**Location:** `grub-core/osdep/linux/ofpath.c`,
`of_find_fc_host()` line ~516 and
`of_path_get_nvme_nsid()` line ~679

In `of_find_fc_host()`, `fopen(node->filename, "r")` is called
without checking the return value. If the file cannot be opened
(permissions, race with sysfs removal), `fp` is NULL and the
subsequent `fscanf(fp, ...)` and `fclose(fp)` are undefined
behavior — typically a segfault.

The same pattern appears in `of_path_get_nvme_nsid()`:
`fp = fopen(buf, "r")` followed immediately by
`fscanf(fp, "%u", &(nsid))` with no NULL check.

Note that `of_path_get_nvmeof_adapter_info()` in the same file
correctly checks fopen return values using
`if (!(fp = fopen(...)) || fscanf(...) != 1)`, so the omission
in these two functions is inconsistent.

**Draft fix** for `of_find_fc_host()`:

```diff
--- a/grub-core/osdep/linux/ofpath.c
+++ b/grub-core/osdep/linux/ofpath.c
@@ -516,6 +516,12 @@
       fp = fopen(node->filename, "r");
+      if (!fp)
+        {
+          node = node->next;
+          continue;
+        }
       buf = malloc (sizeof (char) * 512);
```

**Draft fix** for `of_path_get_nvme_nsid()`:

```diff
--- a/grub-core/osdep/linux/ofpath.c
+++ b/grub-core/osdep/linux/ofpath.c
@@ -679,6 +679,13 @@
   fp = fopen(buf, "r");
+  if (!fp)
+    {
+      free (sysfs_path);
+      free (buf);
+      return 0;
+    }
   fscanf (fp, "%u", &(nsid));
```

### Issue 2: Heap buffer overflow via strcat on exactly-sized buffer

**Location:** `grub-core/osdep/unix/platform.c`,
`add_multiple_nvme_bootdevices()`, lines ~207-213

`nvme_get_syspath()` returns a `malloc`'d buffer sized exactly
for the path string (either from
`block_device_get_sysfs_path_and_link()` or `xrealpath()`).
The code then appends to it with `strcat`:

```c
sysfs_path = nvme_get_syspath (nvme_ns);
// ...
strcat (sysfs_path, "/device");     // 7 bytes past allocation
// or
strcat (sysfs_path, "/subsystem");  // 10 bytes past allocation
```

This writes past the allocated buffer, corrupting heap metadata
or adjacent allocations.

**Draft fix** — use `xasprintf` to build the extended path:

```diff
--- a/grub-core/osdep/unix/platform.c
+++ b/grub-core/osdep/unix/platform.c
@@ -207,11 +207,15 @@
   if (grub_strstr (ofpath, "fibre-channel"))
     {
-      strcat (sysfs_path, "/device");
+      char *tmp = xasprintf ("%s/device", sysfs_path);
+      free (sysfs_path);
+      sysfs_path = tmp;
       is_FC = true;
     }
   else
     {
-      strcat (sysfs_path, "/subsystem");
+      char *tmp = xasprintf ("%s/subsystem", sysfs_path);
+      free (sysfs_path);
+      sysfs_path = tmp;
       is_FC = false;
     }
```

### Issue 3: Memory leaks in of_path_of_nvme NVMeoFC path

**Location:** `grub-core/osdep/linux/ofpath.c`,
`of_path_of_nvme()`, lines ~756 and ~800

Two leaks in the NVMeoFC code path (both with-partition and
without-partition branches):

**(a)** `sysfs_path` from `nvme_get_syspath()` is not freed
before being reassigned:

```c
sysfs_path = nvme_get_syspath (nvmedev);   // allocation A
// ...
sysfs_path = of_find_fc_host (...);        // allocation B
// allocation A leaked
```

**(b)** `free(nvmeof_info)` frees the struct but not the three
members allocated by `of_path_get_nvmeof_adapter_info()`:
`host_wwpn`, `target_wwpn`, and `nqn` (each 256 bytes). These
members are used up to the `snprintf(disk, ...)` call, after
which they can be freed.

**Draft fix** (shown for the with-partition branch; same
pattern applies to the without-partition branch):

```diff
--- a/grub-core/osdep/linux/ofpath.c
+++ b/grub-core/osdep/linux/ofpath.c
@@ -756,6 +756,7 @@
+          free (sysfs_path);
           sysfs_path = of_find_fc_host (nvmeof_info->host_wwpn);
@@ -775,6 +776,8 @@
+          free (nvmeof_info->host_wwpn);
+          free (nvmeof_info->target_wwpn);
+          free (nvmeof_info->nqn);
           free (nvmeof_info);
```
