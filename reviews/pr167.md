## Additional findings

The TFTP fix correctly handles the case where a server sends OACK
followed by an error (instead of error immediately after RRQ). The
guard `data->have_oack == 1 && data->block == 0` targets exactly the
post-OACK, pre-data error window. Setting `data->sock = NULL` after
`grub_net_udp_close` prevents `tftp_close()` from sending an error
packet back — `tftp_close` checks `if (data->sock)` before using the
socket.

The HTTP fix adds `data->first_line_recv = 1` in the 404 and default
error paths of `parse_line`. Without this, the first-line state isn't
updated when an error is received, which could cause reprocessing. The
new block in `http_establish` (after the connection wait loop) checks
`data->err` and propagates the HTTP error code — previously, a 404
was silently swallowed and the file appeared to exist.

The `test -f` fix detects network devices (`!dev->disk && dev->net`)
and probes file existence via `grub_file_open` with
`GRUB_FILE_TYPE_GET_SIZE | GRUB_FILE_TYPE_NO_DECOMPRESS`. This is
the right type combination: `GET_SIZE` skips TPM measurement (avoiding
the OOM issue PR166 addresses), and `NO_DECOMPRESS` avoids unnecessary
decompression. The `grub_errno = GRUB_ERR_NONE` clear suppresses the
file-not-found error that `grub_file_open` sets on failure, so the
`test` command returns false without propagating a GRUB error.
