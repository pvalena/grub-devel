## Additional findings

The pre-existing `write_cr0` bug (inline asm writes to `%%cr4`
instead of `%%cr0`) is still present in this branch — MR !140
addresses it separately. This MR's changes don't interact with
that bug: `enable_sse` only calls `write_cr0` when `orig_cr0
!= cr0`, and the reset path does the same. The incorrect
target register is unchanged either way.

The `hwf_enabled` guard (commit 4) prevents a concrete
corruption scenario: without it, a second `enable` call would
re-read the already-modified CR0/CR4 into `old_cr0`/`old_cr4`,
making the subsequent `reset` restore the wrong values.

Reset order (XCR0 → CR4 → CR0) correctly reverses the enable
order — XGETBV/XSETBV require CR4.OSXSAVE to still be set.
