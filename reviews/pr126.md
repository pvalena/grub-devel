# AI Review: MR !126 - efibootnext: Require target boot entry to exist and show its label

**Commits:**
1. **a97584dbb** - efibootnext: Require the target boot entry to exist and show its label

## Issues Found

### Issue 1: Leaked `grub_errno` from best-effort decode causes spurious error message

**Location:** `grub-core/commands/efi/efibootnext.c`, the best-effort label decode block
(lines ~98-123 in the final file)

If `grub_malloc` fails when allocating the `desc` or `label` buffer in the best-effort decode
block, `grub_errno` is set to `GRUB_ERR_OUT_OF_MEMORY` but never cleared. The comment and commit
message both state this decode is "best-effort" and "not fatal", but the leaked `grub_errno`
causes a real problem: after the command returns `GRUB_ERR_NONE`, the script executor
(`grub-core/script/execute.c:1083`) calls `grub_print_error()`, which checks the global
`grub_errno` and prints `"error: out of memory."` to the user -- even though BootNext was
successfully set.

This is a low-memory scenario, so the malloc failure for a small decode buffer is plausible (GRUB
runs with constrained heap). The user would see:

```
BootNext set to Boot0002
error: out of memory.
```

The fix is to clear `grub_errno` after the best-effort decode block, before calling
`grub_efi_set_variable`, so that a non-fatal allocation failure does not leak into the
command's apparent result.

```diff
--- a/grub-core/commands/efi/efibootnext.c
+++ b/grub-core/commands/efi/efibootnext.c
@@ -121,6 +121,10 @@ grub_cmd_bootnext (grub_command_t cmd __attribute__ ((unused)),
         }
     }
 
+  /* The label decode above is best-effort; do not let a failed malloc
+     leak grub_errno into the caller when BootNext is set successfully.  */
+  grub_errno = GRUB_ERR_NONE;
+
   status = grub_efi_set_variable ("BootNext", &global,
                                   &entry, sizeof (entry));
   if (status != GRUB_ERR_NONE)
```
