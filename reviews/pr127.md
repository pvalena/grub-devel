## Issues Found

### Issue 1: Memory leak in `http_receive()` error path

**Location:** `grub-core/net/http.c`, `http_receive()`, chunked transfer block

When `grub_netbuff_put(nb2, data->chunk_rem)` fails, the code returns
`grub_errno` without freeing the just-allocated `nb2`:

```c
nb2 = grub_netbuff_alloc (data->chunk_rem);
if (!nb2)
    return grub_errno;
err = grub_netbuff_put (nb2, data->chunk_rem);
if (err)
    return grub_errno;       // nb2 leaked
```

```diff
--- a/grub-core/net/http.c
+++ b/grub-core/net/http.c
@@ http_receive, chunked transfer block
   err = grub_netbuff_put (nb2, data->chunk_rem);
   if (err)
-    return grub_errno;
+    {
+      grub_netbuff_free (nb2);
+      return grub_errno;
+    }
```

### Issue 2: IA_NA duplicate detection fails for IAID=0

**Location:** `grub-core/net/bootp.c`, `parse_dhcp6_option()`,
case `GRUB_NET_DHCP6_OPTION_IA_NA`

The guard `if (dhcp6->iaid || len < sizeof (*ia_na))` uses a non-zero
`iaid` to detect "already parsed an IA_NA". Since the `dhcp6` struct is
zero-initialized and IAID=0 is a legitimate server-assigned value (GRUB
itself starts IAIDs at 0 in `grub_cmd_bootp6`), the guard cannot
distinguish "not yet parsed" from "parsed with IAID=0". A second IA_NA
option in the same message would pass the guard and overwrite
t1/t2/iaaddr.

```c
if (dhcp6->iaid || len < sizeof (*ia_na))    // iaid==0 means "not parsed" AND "parsed with IAID=0"
```

A boolean flag (`ia_na_parsed`) or a dedicated field would fix this.

### Issue 3: VCI `strncmp` allows prefix matches

**Location:** `grub-core/net/bootp.c`, `grub_net_configure_by_dhcp_ack()`

The Vendor Class Identifier check uses `grub_strncmp` with the option
length as the limit:

```c
if (grub_strncmp ((const char *) opt, "HTTPClient", opt_len) == 0
```

If `opt_len < 10`, this compares only `opt_len` bytes, so any prefix of
"HTTPClient" (e.g., a 4-byte VCI "HTTP") would match. The DHCP option
data is not NUL-terminated. The boot_file would then be passed to
`dissect_url()` which would likely fail on non-URL data.

Fix: add `opt_len == 10 &&` before the `strncmp`.

### Issue 4: DNS option length truncation in IPv4 DHCP packet construction

**Location:** `grub-core/net/drivers/efi/efinet.c`,
`grub_efinet_create_dhcp_ack_from_device_path()`, IPv4 path

The DNS option length is assigned via `*ptr++ = size_dns` where `ptr` is
`grub_uint8_t *` and `size_dns` is `grub_efi_uintn_t`. The DHCP option
length field is 8 bits, so values > 255 silently truncate. With 4-byte
IPv4 addresses, this requires >63 DNS servers -- extremely unlikely but
the `grub_memcpy(ptr, dns, size_dns)` still writes the full untruncated
amount, causing a buffer overrun past the netbuff allocation.

```c
*ptr++ = size_dns;                  // truncates to uint8_t
grub_memcpy (ptr, dns, size_dns);   // copies full size_dns bytes
```

A bounds check capping `num_dns` (or rejecting > 63) before the
allocation would fix this.
