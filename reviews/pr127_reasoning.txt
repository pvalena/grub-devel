== Issue 1: Memory leak in http_receive() error path ==

Discovery:
Commit 8e9a77d95 adds a return-value check for grub_netbuff_put(). The
alloc-put-copy sequence at lines ~291-296:
  nb2 = grub_netbuff_alloc(data->chunk_rem);
  if (!nb2) return grub_errno;           // correct: nothing to free
  err = grub_netbuff_put(nb2, data->chunk_rem);
  if (err) return grub_errno;            // BUG: nb2 not freed

Source trace:
grub_netbuff_alloc() (grub-core/net/netbuff.c) allocates a netbuff from
the heap. On success nb2 is a valid heap allocation. grub_netbuff_put()
adjusts nb2->tail but can fail if the requested size exceeds the buffer.
On that failure path, nb2 is allocated but never passed to
grub_netbuff_free() or grub_net_put_packet().

Consequence:
Each failure leaks one netbuff (header + data buffer). In the chunked
HTTP transfer path, this could happen repeatedly if the remote server
sends chunk sizes that exceed the netbuff capacity. Leaks accumulate
in GRUB's constrained heap.

Fix verification:
Adding grub_netbuff_free(nb2) before the return is correct. nb2 has not
been passed to grub_net_put_packet() yet, so no other code holds a
reference to it. The free is safe and matches the pattern used elsewhere
in the same file (e.g., the nb2 alloc failure path just above).

== Issue 2: IA_NA duplicate detection fails for IAID=0 ==

Discovery:
Reading parse_dhcp6_option() case GRUB_NET_DHCP6_OPTION_IA_NA (line 882).
The guard:
  if (dhcp6->iaid || len < sizeof (*ia_na))
uses dhcp6->iaid as a "already parsed" flag. Since the struct is
zero-initialized (grub_zalloc in callers), iaid starts as 0.

Source trace:
Traced IAID assignment in grub_cmd_bootp6 (line ~1101):
  se->iaid = iaid;
where iaid is a loop counter starting at 0. So GRUB sends IA_NA requests
with IAID=0 for the first interface. The server echoes back IAID=0 in its
IA_NA option. After parsing, dhcp6->iaid = grub_be_to_cpu32(ia_na->iaid)
= 0, which is indistinguishable from "not yet parsed."

If the server response contains a second IA_NA option (permitted by
RFC 3315 Section 22.4), the guard evaluates as:
  if (0 || len < sizeof(*ia_na))  -->  if (len < sizeof(*ia_na))
If the second IA_NA has sufficient length, it passes and overwrites
iaid, t1, t2, and iaaddr from the first IA_NA.

Consequence:
Silent overwrite of DHCPv6 lease parameters (t1, t2, assigned address)
from a duplicate IA_NA option. In practice, servers rarely send duplicate
IA_NAs for the same IAID, so this is low probability but the logic error
is clear.

Fix verification:
Using a boolean flag (e.g., adding an int ia_na_parsed field to the
dhcp6 struct, or a local variable passed through the callback) would
correctly distinguish "never parsed" from "parsed with IAID=0". The
existing iaid field cannot serve double duty as both a value and a flag.

== Issue 3: VCI strncmp allows prefix matches ==

Discovery:
Reading the HTTP Boot DHCPACK processing (commit 14dbf942c). Line 408:
  if (grub_strncmp ((const char *) opt, "HTTPClient", opt_len) == 0

Source trace:
grub_strncmp (grub-core/kern/misc.c) compares at most n bytes, where
n = opt_len. The DHCP option data (opt) is NOT NUL-terminated -- it's
a raw byte sequence of length opt_len from the DHCP packet.

If opt_len = 4 and opt = "HTTP":
  grub_strncmp("HTTP", "HTTPClient", 4) compares 4 bytes: H==H, T==T,
  T==T, P==P -> returns 0 (match).

The code then treats this as an HTTP Boot server and passes boot_file to
dissect_url(). If boot_file is not actually a URL (because the server is
not really an HTTP Boot server), dissect_url will likely fail, but could
produce garbage results if the boot_file happens to contain "://" at some
offset.

Consequence:
A DHCP server with a VCI that is a prefix of "HTTPClient" (e.g., "HTTP",
"HTTPC") would be incorrectly identified as an HTTP Boot server. The
subsequent URL parsing of a non-URL boot_file path would either fail
(likely) or produce a malformed URL (unlikely but possible). Low
probability -- VCI strings are specific identifiers and "HTTP" alone is
not a standard VCI.

Fix verification:
Adding `opt_len == 10 &&` (where 10 = strlen("HTTPClient")) before the
strncmp ensures only exact-length matches succeed. This is correct
because the DHCP option is not NUL-terminated, so length must be checked
explicitly. The constant 10 could also be written as
`sizeof("HTTPClient") - 1` for clarity.

== Issue 4: DNS option length truncation (IPv4 path) ==

Discovery:
Reading grub_efinet_create_dhcp_ack_from_device_path() IPv4 DNS option
construction (commit 32da12a92, lines ~602-618):
  grub_efi_uintn_t size_dns = sizeof(*dns) * num_dns;
  ...
  *ptr++ = size_dns;               // ptr is grub_uint8_t *
  grub_memcpy(ptr, dns, size_dns);

Source trace:
ptr is grub_uint8_t *, so the assignment truncates size_dns to 8 bits.
The DHCP option length field (RFC 2132) is a single octet, max 255.
With sizeof(grub_efi_ipv4_address_t) = 4, truncation occurs when
num_dns > 63 (64 * 4 = 256, truncated to 0).

The critical part: grub_netbuff_put(nb, size_dns + 2) allocates space
for the full size_dns, and grub_memcpy copies size_dns bytes. But the
option header claims only (size_dns & 0xFF) bytes. If size_dns > 255,
the memcpy writes past what the option header declares, and subsequent
option parsing (by any DHCP option reader) would misinterpret the data
as a new option starting at the wrong offset.

Note: the IPv6 path (line 744) uses opt->len = grub_cpu_to_be16(size_dns)
which is a 16-bit field (DHCPv6 option format) -- no truncation issue.

Consequence:
With >63 DNS servers from firmware (extremely unlikely), the constructed
DHCP packet would have a corrupted option chain. The option length byte
would be wrong, and any code parsing subsequent options in the packet
would read garbage.

Fix verification:
Capping num_dns at 63 (or the equivalent size_dns <= 255 check) before
the allocation is correct. Alternatively, if >63 DNS servers should be
supported, the code would need to split across multiple DHCP options
(RFC 2132 allows repeated options). In practice, a simple cap is the
right fix since >63 DNS servers is not a realistic configuration.
