## Additional findings

The refactoring replaces `__x86_64__` / `__i386__` with
`GRUB_CPU_SIZEOF_VOID_P` for 32/64-bit code path selection
throughout `multiboot.c`, `multiboot_mbi2.c`, and `multiboot_elfxx.c`.
This correctly generalizes without changing behavior on existing x86
and MIPS targets.

The intermediate `grub_uint32_t mbi_value` in `grub_multiboot_boot()`
is necessary because `MULTIBOOT_MBI_REGISTER` now expands to
architecture-specific register fields of different widths (32-bit
`r[1]` on ARM, 64-bit `x[1]` on ARM64/RISC-V). The assignment
`state.MULTIBOOT_MBI_REGISTER = mbi_value` zero-extends on 64-bit
platforms, which is correct for multiboot2's 32-bit MBI addresses.

The ELF loader conditionalization using `MULTIBOOT_ELF32_MACHINE` /
`MULTIBOOT_ELF64_MACHINE` correctly limits each architecture to
its supported ELF format: ARM defines only ELF32, ARM64 and RISC-V 64
define only ELF64, RISC-V 32 defines only ELF32. The `#error "Please
complete this"` in `multiboot_elfxx.c` and `multiboot_mbi2.c` is
replaced with architecture-neutral code.

The relocator selection excludes x86_64 from the generic 64-bit path
(`GRUB_CPU_SIZEOF_VOID_P == 8 && !defined(__x86_64__)`), which is
correct because x86_64 uses its own EFI relocator path to transition
from 64-bit to 32-bit protected mode. Similarly, the 32-bit EFI boot
path excludes i386 (`GRUB_CPU_SIZEOF_VOID_P == 4 && !defined(__i386__)`)
because i386 has its own EFI relocator.

Each new architecture provides: a relocator (C + assembly), an
uppermem stub, a multiboot header with register assignments matching
the architecture's calling convention, a memory header, and a
relocator header.
