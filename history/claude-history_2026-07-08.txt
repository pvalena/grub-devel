  Phase 1: Group into series

  Patches belong together as a series when they share the same subject stem (e.g., commands/ls:) and the same version tag (v3, v4, etc.) with sequential N/M numbering. The key
  grouping signals from failed_subj.txt:

  - [PATCH vN M/T] — version N, part M of T
  - Same topic prefix (e.g., appended signatures, tpm2_key_protector, lvm)
  - Consecutive branch IDs within the same month

  Approach: Parse failed_subj.txt to extract (topic, version, M, T) tuples, group by (topic, version, T). A series = all patches sharing the same group key.

  Special cases:
  - Re: prefixed entries (143 total) — these are replies/followups, not patches. Likely to be reviewed as discussion, not code. Separate bucket.
  - [SECURITY PATCH], [RESEND PATCH], [DRAFT PATCH], Subject: prefixed — non-standard tags, handle individually.

  Phase 2: Identify cross-version duplicates

  Same series submitted as v2, v3, v4, etc. — keep latest only (per project principle). Example already visible:
  - commands/ls v3 (0021-0026), v4 (0045-0050), v5 (0002-0007), v6 (0012-0018) — keep v6 only
  - tpm2_key_protector v2 (0053-0063), v3 (0076-0085), v4 (0136-0147) — keep v4 only
  - appended signatures v2, v4, v5, v7, v8... — keep latest version

  Phase 3: Cross-reference with existing branches

  The 26 branches in grub/ that are NOT in failed.txt — these are the original Phase 1 branches (cleanly applied the first time around). Check if any failed patch subjects
  match these already-merged/reviewed branches. A failed patch for the same topic at a later version might supersede an existing branch.

  Phase 4: Triage the CONFLICT bucket

  For the ~968 conflict branches:
  1. Series-complete vs partial: If only some patches in a series have conflicts and others are ALREADY_APPLIED, the series is partially merged — flag for manual review.
  2. FAILED.patch analysis: The FAILED.patch file in each commit shows the intended change. Can be used for diff-based dedup against master or other branches.
  3. Re: entries: Classify as discussion-only (no code) vs. actual revised patches.

  - import/unique_failed.txt — the final list of branches worth investigating (after removing already-applied and cross-version duplicates)
